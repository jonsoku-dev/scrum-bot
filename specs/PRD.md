# AI-Augmented Scrum Bot: PRD (기획서)

> Slack 대화와 회의록을 기반으로 Jira 티켓 초안을 생성하고 의사결정을 자동 기록하는 AI 어시스턴트 시스템

## 1. 문제 정의 (Failure Modes)
현대적 애자일 스크럼 환경에서 발생하는 5가지 핵심 비효율을 해결합니다.
1. **인력 부족**: 기획/QA/디자인 등 다양한 직군이 모든 스크럼에 실시간 참여하기 어려움.
2. **커뮤니케이션 파편화**: Slack에서 중요한 결정이 내려지지만 Jira 반영은 누락되거나 지연됨.
3. **회의 밀도 저하**: 1시간 회의 중 상당 시간이 "정보 공유"와 "티켓 정리"에 소비됨.
4. **컨텍스트 유실**: "왜 그렇게 결정했는지"에 대한 히스토리가 휘발되어 동일한 논쟁이 반복됨.
5. **책임 소재 불명확**: 업무량 증가로 인해 티켓 관리 책임이 흐려지고 정합성이 깨짐.

## 2. 목표 및 KPI
### 핵심 목표
- 의사결정에서 실행(Jira 반영)까지의 번역 비용 최소화.
- 비참석 직군(Biz/QA/Design)의 관점을 AI를 통해 선제적으로 보강.
- 모든 결정의 근거(Source)를 추적 가능한 상태로 유지.

### Phase 0 성공 지표 (MVP)
- **Slack 메시지 수집 성공률**: ≥ 0.99 (시스템 안정성)
- **요약 생성 P95 Latency**: ≤ 30s (사용자 경험)
- **대시보드 접속 성공률**: 100% (가용성)

### Phase 1+ 성공 지표 (Production)
- **결정 일치율 (Decision Fidelity)**: AI 요약과 실제 합의 내용 일치도 ≥ 0.95
- **티켓 자동화율**: 신규 티켓 중 AI 초안 기반 생성 비율 ≥ 0.60
- **회의 밀도**: 회의 안건 중 "의사결정 필요" 항목 비중 ≥ 0.70

## 3. 핵심 원칙
1. **AI는 비서이지 결정권자가 아니다**: 모든 실행은 반드시 인간의 승인(HITL)을 거친다.
2. **모든 출력에는 출처가 있다**: 근거(Slack 링크, 회의록 ID) 없는 생성/수정은 금지한다.
3. **최신성이 진실이다**: 오래된 정보는 자동으로 영향력을 잃도록 설계한다 (Temporal Decay).
4. **통합은 계약이다**: Slack/Jira/LLM 간의 데이터 교환은 명시적인 Zod 스키마로 고정한다.
5. **안전 설계**: 무한 루프와 비용 폭주를 시스템적으로 차단한다 (Max Iteration, Timeout).

## 4. 페르소나 및 AI 역할
| 페르소나 | 주요 니즈 | AI가 대체/보조하는 역할 | 인간이 유지하는 역할 |
|----------|-----------|-------------------------|----------------------|
| **기획자** | 정리 비용 절감 | 대화 요약, 티켓 초안 작성 | 최종 의사결정, 우선순위 확정 |
| **개발자** | 명확한 AC(수락 조건) | 요구사항 상세화, 히스토리 검색 | 코드 구현, 기술적 검토 |
| **Biz** | ROI 근거 확보 | 비즈니스 임팩트 분석 제안 | 사업적 판단, 예산 승인 |
| **QA** | Edge Case 파악 | 테스트 케이스 및 사이드 이펙트 제안 | 최종 품질 승인, 정책 수립 |
| **운영자** | 시스템 관리 | 감사 로그 기록, 비용 모니터링 | 권한 승인, 모델 정책 변경 |

## 5. 유즈케이스
- **UC-00: [Phase 0] 실시간 수집 및 요약**
  - 특정 채널의 메시지를 실시간 수집하고 `/summarize` 커맨드로 즉시 요약 제공.
- **UC-01: [Phase 1] Slack 결정 감지 및 Jira 반영**
  - 특정 키워드나 리액션 발생 시 결정 카드 생성 및 승인 후 Jira 티켓 자동 생성.
- **UC-02: [Phase 1] 회의록 Action Items 추출**
  - 업로드된 회의록에서 할 일을 추출하여 Jira Draft 목록으로 변환.
- **UC-03: [Phase 2] 멀티에이전트 리뷰**
  - 기획안에 대해 Biz/QA/Design 에이전트가 입체적 리뷰 및 중재안 제안.
- **UC-04: [Phase 2] Jira-Slack 양방향 동기화**
  - Jira 상태 변경 시 Slack 채널에 맥락을 포함한 요약 알림 게시.

## 6. 범위 (In/Out)
### In Scope
- NestJS 기반 단일 프로세스 봇 (Slack Bolt + REST API).
- React Router v7 기반 단일 페이지 대시보드.
- LangGraph.js 기반 AI 워크플로우 오케스트레이션.
- PostgreSQL 기반 데이터 저장 및 감사 로그.

### Out of Scope
- 인간의 승인 없는 Jira 완전 자동 수정.
- 소스 코드 레벨의 자동 리팩토링.
- 다국어 지원 (초기 버전 한국어/영어 전용).

## 7. 리스크 관리
1. **환각 (Hallucination)**: 근거 없는 티켓 생성 리스크 → **인간 승인 강제**.
2. **컨텍스트 오염**: 과거 데이터를 현재 데이터로 오인 → **시간 가중치(Decay) 적용**.
3. **보안/권한**: 토큰 유출 및 권한 오남용 → **최소 권한 원칙(Least Privilege)**.
4. **스팸화**: 과도한 Slack 알림 → **알림 요약 및 채널 분리 정책**.
5. **비용 폭주**: LLM 토큰 과다 사용 → **사용량 제한(Quotas) 및 모니터링**.
6. **계약 파손**: Jira 필드 변경으로 인한 에러 → **Canonical 스키마 매핑 도입**.
7. **아키텍처 복잡성**: 과도한 엔지니어링으로 인한 일정 지연 → **심플 아키텍처(Phase 0 집중) 지향**.

## 8. 단계별 로드맵
- **Phase 0 (1주)**: 인프라 셋업, Slack Bolt 연결, 메시지 저장소 구축, 단순 요약 커맨드 구현.
- **Phase 1 (2주)**: 의사결정 감지 엔진, Draft CRUD, HITL 승인 플로우, Jira 연동.
- **Phase 2 (4주)**: LangGraph 멀티에이전트 리뷰, pgvector 기반 지식베이스, Slack Block Kit 고도화.
- **Phase 3 (지속)**: 품질 평가 자동화, 비용 최적화, 감사/보안 강화.
